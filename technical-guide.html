Your Project Map

Every file in your project has a role. Here's what matters and what each piece does. You can ignore everything else (especially node_modules/ — that's library code Astro needs internally).

src/ — all your source code lives here
pages/ — every file here becomes a URL on your site
index.astro — your homepage
about.astro — your about page
posts/ — each .md file here becomes an article
layouts/ — templates that wrap your content
Base.astro — header + footer + dark mode (wraps every page)
Post.astro — the article template (wraps each post)
styles/
global.css — all your visual design lives in this single file
public/ — images, favicon — served as-is to visitors
images/ — put your article images here
package.json — project name + dependencies (rarely touch this)
astro.config.mjs — Astro settings (rarely touch this)
Analogy

Think of it like a building. layouts/ is the architecture — walls, roof, doors. pages/ is the furniture and art you put inside each room. styles/ is the paint, textures, and lighting. public/ is the stuff visitors can pick up and hold.

The Mental Model

Your site is built with Astro. There's really only one big idea to internalize:

Core idea

Files equal pages. If you create a file at src/pages/letters/first.md, your site will have a page at yoursite.com/letters/first. No configuration needed. The folder structure is the site structure.

There are two types of files you'll work with:

.md files (Markdown) — your content

This is where you write your articles. Markdown is a simple text format: **bold** for bold, *italic* for italic, ## Heading for headings, > quote for blockquotes. You already know this from your posts.

.astro files — your templates and pages

These mix HTML (structure), CSS (visual design), and JavaScript (behavior). Your homepage index.astro is an .astro file because it does more than display text — it queries all your posts and renders them as a list. You generally won't need to edit these directly — that's what you have me for.

CSS Variables: Your Design Control Panel

At the top of your global.css file, there's a block of named values. These are CSS variables — named design tokens that are referenced everywhere else in the file. Change the value once, and it updates across the entire site.

global.css :root {
  --accent: #D4A017; /* the gold color used everywhere */
  --font-display: 'Instrument Serif'; /* headlines */
  --font-body: 'Inter Tight'; /* body text */
  --cream: #F5F0E8; /* background color */
  --max-width: 860px; /* how wide the content area is */
}

5 Things That Expand What's Possible

Click each card to expand it.

1 Frontmatter is your content database
▶

The --- block at the top of each markdown file is called frontmatter. It's structured data about the post — distinct from the post content itself.

Right now your posts have title, subtitle, date, and tag. But you can add any fields you want, and your site can use them to filter, sort, and display posts differently.

Ando.md ---
title: "New Ontologies: Ando"
subtitle: "Inside Ando"
date: 2026-02-15
tag: company building, future of work

featured: true
author: "Nicole Seah"
coverImage: "/images/ando-cover.jpg"
series: "Founders"
status: "draft"
---

What this unlocks: A "Featured" section on the homepage. Cover images for posts. A draft system where posts with status: draft are hidden from the live site. Filtering posts by series or author.

2 Multiple collections and page types
▶

Right now you only have posts/. But you can create as many content types as you want — each with its own visual personality.

Possible new folders src/pages/
  posts/ ← long-form articles (what you have now)
  notes/ ← short-form observations, links, fragments
  letters/ ← a newsletter archive
  library/ ← a curated reading list
  glossary/ ← a vocabulary of new ontologies

Each collection can have its own layout — different typography, different density, different energy. A glossary might be a single page with alphabetized terms. Notes might be short and dense. The site structure mirrors the intellectual structure of your publication.

3 Reusable components
▶

A component is a reusable visual building block. Instead of writing raw HTML every time you want a styled image with a caption, you create a component once and reuse it everywhere.

What you'd write in a post <!-- Instead of raw HTML: -->
<Figure src="/images/ando-office.jpg" caption="Jackson Square, morning light" />

<Pullquote author="Sara Du">
  I was so raw as a 19-year-old, I just wanted to win.
</Pullquote>

<AuthorCard name="Sara Du" role="CEO, Ando" />

This is how you replace those [picture of product] placeholders in your Ando article with something rich and consistent. You define the look once, then just fill in the details each time.

4 Dynamic content from structured data
▶

Your homepage already does this — it reads all posts, sorts them by date, and lists them automatically. You never have to manually update the homepage when you publish a new article. This pattern is very flexible:

  • → Group posts by tag on an archive page
  • → Show "related posts" at the bottom of each article
  • → Auto-generate a page for each tag (e.g., yoursite.com/tags/company-building)
  • → Create an RSS feed so readers can subscribe
  • → Build a search page that filters posts by keyword
  • → Show a "latest post" banner on the about page

The key insight: your content's frontmatter is queryable. Anything you put in that --- block can be used to organize, filter, and display your content in new ways — without changing the content itself.

5 Client-side interactivity
▶

The <script> blocks in your site are JavaScript that runs in the reader's browser. You already have two: the pixel-reveal animation on your hero image, and the dark mode toggle.

This is what makes a site feel alive versus static. Things you could add:

  • → Reading progress bar — a thin line at the top showing how far through an article someone is (like this guide has)
  • → Scroll animations — sections fade in gently as you scroll down
  • → Active table of contents — highlights the current section as you read
  • → Expandable footnotes — hover or click to reveal a note without leaving the paragraph
  • → Copy-to-clipboard for quotes or links
  • → Email signup form — collect subscriber emails

This guide itself demonstrates several of these: the progress bar, the sidebar highlighting, the expandable cards, and the interactive color demo above. All of these are achievable in your project.

How to Prompt Claude

You don't need to know how to code. You need to know how to describe what you want precisely. Here's the difference:

Be specific about where and what

Vague
"Make it look better"
Specific
"Make the post list on the homepage feel more spacious — more padding between items"

Reference what you see, not the code

Vague
"Fix the typography"
Specific
"The subtitle on article pages feels too close to the title — add more space between them"

Describe the feeling, not just the feature

Thin
"Add a hover effect to post links"
Rich
"When hovering on a post title, I want something subtle and warm — maybe the gold accent color, and a slight shift. Nothing flashy."

Screenshots are powerful

You can drop a screenshot into the chat and say "make it look like this" or "this spacing feels off." Claude can read images and map what you see to the code that controls it.

Ask what's possible before deciding

Great prompts
"What are my options for displaying images in articles?"
"Could the homepage have a featured post that looks different from the rest?"
"Is it possible to have footnotes that expand on hover?"
